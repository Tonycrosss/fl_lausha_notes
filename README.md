# Telegram Broadcast Bot MVP

MVP Telegram-бота для рассылки материалов подписчикам авторов. Пользователи подтверждают подписку через бота, автоматически получают подписку на всех активных авторов, а администратор управляет авторами и планирует рассылки прямо из Telegram.

## Стек

- Python 3.11+
- aiogram 3.x
- SQLite
- APScheduler
- python-dotenv

## Реализовано

- регистрация пользователя через `/start`
- подтверждение подписки одной кнопкой
- защита от дублей пользователей и подписок
- автоматическая подписка пользователя на всех активных авторов
- хранение авторов в SQLite
- админ-панель внутри Telegram по команде `/admin`
- добавление нового автора
- сохранение названия канала и ссылки канала у автора
- редактирование названия канала и ссылки канала
- активация и отключение автора
- одна рассылка может включать нескольких авторов
- создание рассылки через FSM
- загрузка одного или нескольких документов через Telegram `file_id`
- предварительное уведомление перед рассылкой
- отложенная отправка материалов по расписанию
- восстановление задач APScheduler после перезапуска
- статистика по пользователям, авторам и последней выполненной рассылке
- команда `/help`

## 1. Создание бота через BotFather

1. Откройте Telegram и найдите `@BotFather`.
2. Отправьте команду `/newbot`.
3. Задайте имя бота.
4. Задайте username бота, который заканчивается на `bot`.
5. Скопируйте полученный токен.
6. Этот токен укажите в `.env` как `BOT_TOKEN`.

## 2. Как получить `ADMIN_TELEGRAM_ID`

Самый простой способ:

1. Напишите в Telegram боту `@userinfobot` или аналогичному боту, который показывает ваш ID.
2. Скопируйте числовой Telegram ID.
3. Укажите его в `.env` как `ADMIN_TELEGRAM_ID`.

## 3. Настройка `.env`

Создайте файл `.env` рядом с `main.py`:

```env
BOT_TOKEN=123456:ABCDEF
ADMIN_TELEGRAM_ID=123456789
TIMEZONE=Europe/Moscow
```

`TIMEZONE` используется и для парсинга дат из админ-панели, и для APScheduler.

## 4. Установка зависимостей

Рекомендуется использовать виртуальное окружение:

```bash
python3.11 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
```

Если на сервере нет `pip` или `venv`, сначала установите системные пакеты. Для Debian/Ubuntu это обычно:

```bash
sudo apt-get update
sudo apt-get install -y python3-pip python3-venv
```

Если удобнее использовать `uv`:

```bash
uv venv .venv
UV_CACHE_DIR=/tmp/uv-cache uv pip install --python .venv/bin/python -r requirements.txt
```

## 5. Запуск проекта

Локальный запуск:

```bash
python main.py
```

При первом запуске автоматически будет создан файл базы данных `app.db`.

## 6. Переменные окружения

- `BOT_TOKEN` - токен Telegram-бота из `@BotFather`
- `ADMIN_TELEGRAM_ID` - Telegram ID единственного администратора
- `TIMEZONE` - таймзона для парсинга времени и планировщика

Пример:

```env
BOT_TOKEN=123456:ABCDEF
ADMIN_TELEGRAM_ID=1095242020
TIMEZONE=Europe/Moscow
```

## 7. Пользовательский сценарий

1. Пользователь отправляет `/start`.
2. Бот предлагает подтвердить подписку.
3. После подтверждения пользователь помечается как подписанный.
4. Бот автоматически подписывает его на всех активных авторов.
5. Бот показывает список активных авторов.
6. Пользователь получает:
   - уведомление о будущей рассылке
   - материалы в запланированное время

Отписка в MVP не реализована.

## 8. Админ-панель

Администратор должен использовать Telegram ID, указанный в `.env`.

Вызов панели:

```text
/admin
```

В меню доступны кнопки:

- `Авторы`
- `Новая рассылка`
- `Запланированные рассылки`
- `Статистика`

### Авторы

- просмотр списка авторов
- добавление нового автора
- сохранение названия канала и ссылки на канал
- редактирование названия канала
- редактирование ссылки канала
- отключение автора
- повторная активация автора

Если активируется новый автор, все уже подписанные пользователи автоматически получают подписку и на него.

При создании автора администратор вводит:

1. имя автора
2. название канала
3. ссылку на канал

Поддерживаются ссылки вида:

- `https://t.me/...`
- `@username`

### Создание рассылки

FSM-поток:

1. выбрать одного или нескольких авторов
2. ввести название рассылки
3. загрузить 1 или несколько документов
4. нажать `Готово`
5. ввести дату и время уведомления
6. ввести дату и время отправки
7. подтвердить создание

Формат даты и времени:

```text
YYYY-MM-DD HH:MM
```

Пример:

```text
2026-04-20 19:30
```

### Запланированные рассылки

- просмотр всех рассылок со статусами `scheduled` и `notified`
- отображение списка авторов, включенных в конкретную рассылку
- отмена запланированной рассылки

### Что получает пользователь

В уведомлениях и при отправке материалов бот показывает:

- список авторов рассылки
- кликабельные названия каналов вместо сырых ссылок
- затем сами документы

### Статистика

Для MVP отображаются:

- общее количество пользователей
- количество активных авторов
- количество запланированных рассылок
- количество успешно отправленных сообщений в последней выполненной рассылке
- количество ошибок отправки в последней выполненной рассылке

## 9. Схема БД

Используются таблицы:

- `users`
- `authors`
- `user_authors`
- `broadcasts`
- `broadcast_authors`
- `broadcast_files`
- `broadcast_logs`

Хранение сделано через `aiosqlite` и легкий repository layer без ORM.

- `users` хранит Telegram-пользователей и флаг подписки
- `authors` хранит авторов, их активность, название канала и ссылку на канал
- `user_authors` хранит связи пользователей с авторами
- `broadcasts` хранит запланированные рассылки
- `broadcast_authors` хранит связи рассылки с одним или несколькими авторами
- `broadcast_files` хранит Telegram `file_id` и метаданные документов
- `broadcast_logs` хранит результат отправки по каждому пользователю

Статусы рассылки:

- `scheduled`
- `notified`
- `sent`
- `canceled`

## 10. Поведение планировщика

При запуске приложения:

- загружаются все рассылки со статусами `scheduled` и `notified`
- для них восстанавливаются задания планировщика
- если `notify_at` уже прошло, а `send_at` еще нет, планируется только отправка
- если `send_at` тоже прошло, автодогон не выполняется, в лог пишется предупреждение

## 11. Хранение файлов

Для MVP файлы не сохраняются на диск.

Админ загружает документы в Telegram-бота, а в БД сохраняются:

- `telegram_file_id`
- имя файла
- MIME-тип

При рассылке используется повторная отправка через сохраненный `file_id`.

## 12. Деплой на VPS через systemd

Пример директории:

```text
/opt/telegram-broadcast-bot
```

Пример шагов:

```bash
sudo mkdir -p /opt/telegram-broadcast-bot
sudo chown -R $USER:$USER /opt/telegram-broadcast-bot
cd /opt/telegram-broadcast-bot
python3.11 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
```

Создайте файл сервиса:

```bash
sudo nano /etc/systemd/system/telegram-broadcast-bot.service
```

Содержимое:

```ini
[Unit]
Description=Telegram Broadcast Bot
After=network.target

[Service]
Type=simple
User=www-data
WorkingDirectory=/opt/telegram-broadcast-bot
EnvironmentFile=/opt/telegram-broadcast-bot/.env
ExecStart=/opt/telegram-broadcast-bot/.venv/bin/python /opt/telegram-broadcast-bot/main.py
Restart=always
RestartSec=5

[Install]
WantedBy=multi-user.target
```

Запуск и автозапуск:

```bash
sudo systemctl daemon-reload
sudo systemctl enable telegram-broadcast-bot
sudo systemctl start telegram-broadcast-bot
sudo systemctl status telegram-broadcast-bot
```

Просмотр логов:

```bash
journalctl -u telegram-broadcast-bot -f
```

## 13. Как обеспечить постоянную работу

- запускать бота через `systemd`
- использовать `Restart=always`
- хранить `.env` на сервере вне публичного доступа
- регулярно проверять логи через `journalctl`
- делать резервную копию файла `app.db`

## 14. Git и безопасность

- не коммитьте `.env`
- не коммитьте `.venv`
- не коммитьте `app.db`, `app.db-shm`, `app.db-wal`
- перед публикацией репозитория перевыпускайте токен, если он хоть раз был отправлен в чат или попал в историю shell
